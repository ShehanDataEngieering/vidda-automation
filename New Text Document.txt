⚠️ Issues & Gaps
1. Single training_plans row carries entire pipeline state
The plan has one row per pipeline run with risk_matrix, amlr_mappings, training_plan all as JSONB on the same row. Problems:
- 
No versioning — if a user re-runs Step 3 after editing risk scores, you lose history
- 
No audit trail per step — brief mentions "audit-ready" — who overrode which score and when?
- 
Concurrent edits unsafe — two reviewers on the same plan race
Suggest: Add a plan_events table (event-sourced) capturing each step + override with timestamp + reviewer. Or per-step tables: plan_risk_assessments, plan_amlr_mappings, plan_versions.
2. Missing assignments table
Branch 5 (LMS) mentions a table view with status + due dates, but the schema only has training_plans. Where do assignments live?
Suggest: Add plan_assignments table:
plan_assignments (
  id, plan_id, user_id, module_index, quarter,
  due_date, status (not_started|in_progress|completed),
  completed_at
)
3. No regeneration logic after human gates
Brief Section 7 states: "AI regenerates AMLR mapping and training plan using the corrected scores" after Gate 1. Plan doesn't show this loop.
Suggest: After Gate 1 override → trigger Step 4 regeneration. After Gate 2 override → trigger Step 5 regeneration. Need explicit endpoints like POST /api/pipeline/:id/regenerate-from-step/3.
4. RAG retrieval only at Step 4 — Step 5 generates blind
Step 5 (training plan generation) gets AMLR mappings as input but doesn't retrieve actual article text again. The AI will hallucinate why_included justifications without the actual regulation text in context.
Suggest: Step 5 should also retrieve the relevant AMLR chunks (or pass them through from Step 4's RAG retrieval).
5. No quality scoring or validation
The old system had qualityScore.ts (citation grounding check). New pipeline has zero validation:
- 
Are the AMLR mappings using real articles (9-15) and not hallucinated ones (e.g., "Article 42")?
- 
Does every module have all 5 required fields populated?
- 
Does each why_included actually reference the cited amlr_article?
Suggest: Add services/pipelineValidator.ts — runs after each AI call, returns warnings to UI.
6. EUR-Lex scraping reliability not addressed
Branch 0 says "Fetch from EUR-Lex / paste text". EUR-Lex doesn't have a reliable API for full article extraction. The existing htmlChunker.ts targets legislation.gov.uk which won't have AMLR (post-Brexit).
Suggest: Concrete fallback plan — download the official AMLR 2024/1624 PDF from EUR-Lex once, commit it to repo (or seed script), extract Articles 9-15 using existing chunkPdf(). Don't depend on runtime scraping.
7. No structured AI output enforcement
Plan describes JSON outputs but doesn't mention:
- 
response_format: { type: 'json_object' } for non-streaming calls
- 
JSON validation + retry logic (we already built this for quizzes)
- 
Zod schemas for parsing AI responses
Suggest: Define Zod schemas matching the brief's JSON outputs. Validate after each AI call. Retry once on failure.
8. Frontend state management strategy missing
6 screens passing pipeline state between each other. Current app uses useState in App.tsx for screen transitions. With 7 steps, state grows complex:
- 
Where does role_profile live during the session?
- 
What if user refreshes page mid-pipeline?
- 
Does the URL reflect the current step?
Suggest: Decide upfront: (a) URL-based routing per step using react-router, (b) all state from planId in DB, refetched per screen, or (c) Zustand/context store. Pick one before building.
9. 5-role enforcement not in plan
The brief mandates 5 specific roles (MLRO, AML DDI, KYC, TM, Customer Advisor). The plan accepts free-text role descriptions. What if user pastes "Software Engineer"?
Suggest: Step 2 (role analysis) should classify the input into one of the 5 known roles OR flag as "outside scope". This protects the 25% Explainability score.
10. No demo path / happy path defined
Brief Section 10 says demo is 5 minutes max. The plan is comprehensive but doesn't identify the demo flow:
- 
Which role description to paste in demo?
- 
Which AI overrides to show at Gates 1/2/3?
- 
What's the "money shot" — the per-module justification panel?
Suggest: Add a Branch 0 deliverable: demo-script.md with exact role description text + reviewer narrative.

Q2: Versioning — My Recommendation
Hybrid: version integer + plan_events audit log.
training_plans (
  id, company_id, role_title, role_description, line_of_defence,
  risk_matrix JSONB, amlr_mappings JSONB, training_plan JSONB,
  current_step VARCHAR(20),    -- 'role'|'risk'|'amlr'|'plan'|'approved'
  version INTEGER DEFAULT 1,   -- bumps on each step regeneration
  status VARCHAR(20),
  created_at, updated_at
)
plan_events (
  id, plan_id, version, step VARCHAR(20),
  action VARCHAR(30),          -- 'ai_generated'|'human_override'|'approved'|'regenerated'
  reviewer VARCHAR(255),
  before_state JSONB,          -- what changed FROM
  after_state JSONB,           -- what changed TO
  note TEXT,
  created_at
)
Why this:
- 
training_plans always holds the current state — fast read, simple UI
- 
plan_events is the audit trail — satisfies "audit-ready" requirement from brief (25% Explainability)
- 
version integer prevents concurrent edit races (optimistic locking: UPDATE WHERE version = $expected)
- 
On any human gate override → write event row, bump version, regenerate downstream steps
- 
Demo: "Click any module → see WHO changed WHAT and WHEN" → strong jury signal
Simpler than full event-sourcing (no replay logic needed) but captures the audit story.
Q3: Role Validation — Reading the Documents
From vidda_AI_context_brief.md Section 3, the 5 mandatory roles are:
- 
Customer Advisor (1LoD frontline)
- 
KYC Analyst (1LoD specialist)
-

-

-

From New Text Document.txt: roles in scope carry obligations under AMLR + GDPR + FCA + Consumer Duty + UK sanctions.
My recommendation: Soft classification with explicit confirmation.
Step 2 (Role Analysis) AI call should:
1. 
Read pasted role description
2. 
Classify against the 5 known role archetypes (return best match + confidence)
3. 
If high confidence (>0.7) → proceed with that classification
4. 
If low confidence → show user a "Closest match: KYC Analyst (54% confidence). Looks like this might not be an AMLR-scoped role. Continue anyway or paste a different description?" dialog
5. 
Never hard-reject — jury demo might paste anything; soft warning is safer
Schema addition:
interface RoleProfile {
  role_title: string;
  line_of_defence: '1LoD' | '2LoD';
  classified_as: 'Customer Advisor' | 'KYC Analyst' | 'TM Analyst' | 'AML DDI Manager' | 'MLRO' | 'other';
  classification_confidence: number; // 0-1
  daily_activities: string;
  key_decisions: string;
  mistake_consequences: string;
}
This protects the Explainability score — the AI's classification is itself shown as a justified decision.
